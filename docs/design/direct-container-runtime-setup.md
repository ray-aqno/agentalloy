# Design: Direct Container Runtime Setup

## 1. Overview

This design replaces the container deployment path in `agentalloy setup` — currently a
compose-heavy four-service topology (ollama, ollama-pull, agentalloy-init, agentalloy) —
with a single-container, direct podman/docker runtime flow. The new flow uses direct
`podman`/`docker` commands (build, run, volume, inspect) with no compose dependency.

Ollama is installed programmatically inside the container at first boot via a dynamically
generated entrypoint wrapper script (not committed to the repo). The embedder (Ollama) and
datastore (Kuzu) both live inside the single container. All deployments are CPU-only.

The shared setup front-end (hardware detection, pack selection, review summary) remains
common to both native and container paths. Container mode uses fixed values for native-only
prompts: runner=ollama, port=47950, mode=manual, harness=manual.

## 2. Architecture

### 2.1 High-Level Flow

```
User runs: agentalloy setup [--deployment container]
    |
    v
[SHARED PHASE]  -- common to native and container
  1. Hardware detection (detect.py + host target derivation)
  2. Deployment choice prompt (native vs. container)
  3. Pack selection
  4. Review summary and confirmation
    |
    v
[CONTAINER BRANCH]  -- replaces _run_container_flow()
  1. Detect runtime (podman preferred, docker fallback)
  2. Locate build context (cwd -> parents[4] editable install -> auto-clone)
  3. Container preflight checks (runtime, build assets, port free, name conflicts)
  4. Build image: {runtime} build -t agentalloy:local -f Containerfile <context>
  5. Ensure named data volume: {runtime} volume create agentalloy-data
  6. Ensure ~/.ollama directory exists (for models bind mount)
  7. Generate entrypoint wrapper script to temp file
  8. Run container: {runtime} run --replace -d --name agentalloy <volumes> <env> agentalloy:local
  9. Wait for health endpoint: poll http://localhost:47950/health (300s timeout, exponential backoff)
  10. Record state to install_state
  11. Run verify step
  12. Wire harness (if requested)
```

### 2.2 In-Container Bootstrap Sequence

The Containerfile CMD starts an entrypoint wrapper script generated by the setup script.
The bootstrap sequence runs inside the container:

```
1. Check if Ollama is installed (`which ollama`). If not, download and install it.
2. Start `ollama serve` in background on 127.0.0.1:11434.
3. Poll http://127.0.0.1:11434 until Ollama is ready (30s timeout).
4. Check if embedding model (qwen3-embedding:0.6b) is cached (`ollama list | grep`).
   If not, pull it.
5. Check if /app/.bootstrap-complete exists. If yes, skip to step 9.
6. Run migrations if needed (`python -m agentalloy.migrate`).
7. If AGENTIALLOY_PACKS env var is set, run `install-packs` (before uvicorn starts,
   no lock conflict).
8. Create /app/.bootstrap-complete flag file.
9. Start uvicorn on 0.0.0.0:47950.
10. Trap SIGTERM for graceful shutdown.
```

The entrypoint is idempotent: on subsequent starts, the state flag file causes it to skip
directly to step 9.

### 2.3 Volume Layout

```
/app/
  data/                    # Named volume: agentalloy-data
    ladybug                # Kuzu database (file-level lock)
    skills.duck            # DuckDB vector store
    skills/                # Packed skill definitions
  .ollama/                 # Bind mount: ~/.ollama (host)
    models/
      download/
      manifests/
```

- `agentalloy-data`: Named volume managed by the container runtime.
- `~/.ollama`: Bind mount for model cache — transparent for model management from host.

### 2.4 Kuzu Lock Resolution (Day-2 Operations)

The shared `container_service.py` module provides:
- `is_in_container()` — detects container environment (checks `/.dockerenv` or `/app`)
- `stop_service_in_container()` — finds uvicorn PID, sends SIGTERM, waits 15s, SIGKILL
- `restart_service_in_container()` — restarts uvicorn with same command line, polls /health

This replaces the one-shot container pattern. The single container stays running; the app
process is stopped, the CLI command runs, then the app process is restarted. This is the
same pattern already used for native installs (systemd/launchd stop/restart in
`reembed/cli.py:491-508`), adapted for the container process tree.

The `--no-restart` flag from `reembed` suppresses the restart step for both native and
container modes.

## 6. Sequence

### 6.1 Main Container Setup Flow

```
1. User selects "container" deployment type (or --deployment container)
2. Shared flow runs: hardware detection, pack selection, review
3. CPU-only warning displayed with user confirmation
4. Runtime detection: shutil.which("podman") -> shutil.which("docker")
5. Build context location:
   a. cwd (user ran setup from inside the clone)
   b. parents[4] of __file__ (editable install -> repo root)
   c. Auto-clone to ~/.cache/agentalloy/repo (for uv tool install users)
6. Preflight checks:
   a. Runtime binary available
   b. Build assets present (Containerfile, pyproject.toml, uv.lock)
   c. Port 47950 free
   d. No name conflict (agentalloy container not running)
7. Build image: podman build -t agentalloy:local -f Containerfile <context>
8. Create volume: podman volume create agentalloy-data
9. Create ~/.ollama directory
10. Generate entrypoint wrapper (temp file with bootstrap logic)
11. Run container:
    podman run --replace -d --name agentalloy \
      --volume agentalloy-data:/app/data \
      --volume ~/.ollama:/root/.ollama \
      -e AGENTIALLOY_PACKS={packs} \
      -e ENTRYPOINT=/tmp/entrypoint.sh \
      -p 47950:47950 \
      agentalloy:local
12. Wait for health endpoint (poll /health, 300s timeout, exponential backoff)
13. Record state to install_state
14. Run verify
15. Wire harness (if requested)
16. Display success message with operational commands
```

### 6.2 In-Container Bootstrap Sequence

```
1. Entry script checks: which ollama
   - Not installed: download and install Ollama (platform-specific)
   - Installed: skip to step 2
2. Start: ollama serve & (background on 127.0.0.1:11434)
3. Poll: curl http://127.0.0.1:11434 (30s timeout)
4. Check model cache: ollama list | grep qwen3-embedding
   - Not cached: ollama pull qwen3-embedding:0.6b
   - Cached: skip
5. Check /app/.bootstrap-complete
   - Exists: skip to step 9
   - Missing: continue to step 6
6. Run migrations: python -m agentalloy.migrate
7. If AGENTIALLOY_PACKS set: uv run agentalloy install-packs
8. Touch /app/.bootstrap-complete
9. Start uvicorn: exec uv run uvicorn agentalloy.app:app --host 0.0.0.0 --port 47950
10. Trap SIGTERM for graceful shutdown
```

### 6.3 Day-2 Operation: Reembed in Container

```
1. User runs: podman exec agentalloy uv run agentalloy reembed
2. reembed/cli.py detects container (is_in_container() = True)
3. stop_service_in_container(): finds uvicorn PID, SIGTERM, wait 15s, SIGKILL
4. Verify Kuzu lock released (test connection)
5. Run reembed operation
6. restart_service_in_container(): spawn uvicorn, poll /health (30s timeout)
```

## 7. Error Handling

### 7.1 Runtime Not Found

- Check `shutil.which("podman")` and `shutil.which("docker")`.
- If neither found, display clear error with installation instructions.
- Exit with code 1 (EXIT_USER).

### 7.2 Build Context Not Found

- Search order: cwd -> parents[4] -> auto-clone.
- If all fail, display error with guidance on how to set up a local checkout.
- In non-interactive mode, exit with code 1.

### 7.3 Build Failure

- Capture build output to log file.
- Display last 30 lines on failure.
- Exit with code 1 (EXIT_USER).

### 7.4 Container Start Failure

- Check for port conflicts, volume issues, image availability.
- Display relevant error messages.
- Exit with code 1 (EXIT_USER).

### 7.5 Health Check Timeout

- Poll /health with exponential backoff (2s, 4s, 8s, 16s, 32s, 64s, 128s) up to 300s.
- If timeout, display error with instructions to check container logs.
- Exit with code 1 (EXIT_USER).

### 7.6 Entry Point Script Generation Failure

- Write to temp file with atomic write pattern.
- If write fails, display error and exit.
- Temp file is cleaned up after container start (or on failure).

### 7.7 Volume Creation Failure

- If volume already exists, skip creation (idempotent).
- If creation fails, display error with cleanup instructions.
- Exit with code 1 (EXIT_USER).

## 8. Performance

### 8.1 Image Build

- The image build uses layer caching: dependency manifests are copied first, then source.
- Subsequent builds should be fast if only source code changes.
- Build timeout: 600s (10 minutes) — generous for a Python project.

### 8.2 Health Check

- Exponential backoff avoids hammering the health endpoint.
- 300s timeout is generous for container startup (includes Ollama install + model pull).

### 8.3 Model Pull

- The model pull happens inside the container, not on the host.
- Timeout: 600s (10 minutes) for the model pull.
- Model cache persists in `~/.ollama` bind mount across container rebuilds.

### 8.4 Container Start

- The container start is fast (< 1s) once the image is built.
- Ollama installation happens inside the container at first boot.
- Subsequent starts skip Ollama install (idempotent entrypoint).

## 9. Security

### 9.1 Entrypoint Script

- Generated to a temp file, not committed to the repo.
- Written with restricted permissions (0600).
- Temp file cleaned up after container start.

### 9.2 Volume Permissions

- `agentalloy-data` named volume is owned by the container user.
- `~/.ollama` bind mount uses host directory — user controls access.
- No root escalation needed (rootless podman compatible).

### 9.3 Network Exposure

- Only port 47950 is published to the host.
- Ollama (port 11434) is NOT published — only reachable inside the container.
- No compose network aliases needed (single container).

### 9.4 Auto-Clone

- Auto-clone to `~/.cache/agentalloy/repo` uses HTTPS (no SSH key needed).
- Clone is shallow (`--depth=1`) to minimize attack surface.
- On refresh, hard-reset to origin/main (no local edits preserved).

## 10. Operational Contract (No Compose)

Users interact with the container directly via podman/docker commands:

```
# Logs
podman logs -f agentalloy
docker logs -f agentalloy

# Restart
podman restart agentalloy
docker restart agentalloy

# Inspect
podman inspect agentalloy
docker inspect agentalloy

# Exec into container
podman exec -it agentalloy sh
docker exec -it agentalloy sh

# Volume inspect
podman volume inspect agentalloy-data
docker volume inspect agentalloy-data

# Stop
podman stop agentalloy
docker stop agentalloy

# Remove
podman rm -f agentalloy
docker rm -f agentalloy
```

## 11. Apple Silicon Special Handling

For Apple Silicon hosts, the entrypoint wrapper detects the platform and installs Ollama
via Homebrew (brew install --cask ollama-app) rather than the generic install script.
This is still CPU-only — no GPU passthrough.
