# AgentAlloy Spec: Unified Setup With Direct Container Runtime

## Summary
Unify setup UX across native and container installs while replacing the container compose topology with a direct runtime flow.

Container mode must:
- Expose one reachable AgentAlloy endpoint for harnesses.
- Keep both embedder (Ollama) and datastore (Kuzu) inside the containerized deployment.
- Remain CPU-only for this phase.
- Use direct podman or docker commands only (no compose path).
- Run everything inside a single container — no sidecars, no multi-service orchestration.

## Problem
Current setup diverges early:
- Native path runs full setup configuration (runner, model, hardware, port, mode, packs, harness, upstream).
- Container path branches into a separate compose-heavy flow with 4 services (ollama, ollama-pull, agentalloy-init, agentalloy).

This creates inconsistent UX and increases execution complexity for container installs. The compose dependency adds surface area (compose.yaml, compose providers, project names, network aliases) that is unnecessary for a single-service deployment.

## Decisions
- Reuse existing hardware detection from setup (`detect.py` + host target derivation).
- Keep container runtime CPU-only. Apple Silicon requires special handling (brew + ollama-app install) but still runs CPU-only.
- Keep pack selection in shared setup flow.
- Run everything inside a single container — no sidecar services, no compose.
- Ollama is installed programmatically through the setup script after the basic container is built, not baked into the image.
- Remove compose as a supported install/runtime path for container setup.
- `compose.yaml` remains in the repo as a deprecated file for advanced users but is no longer part of the setup path.

## Non-Goals
- GPU parity between native and container installs.
- Host-managed embedder for container path.
- Backward compatibility with the old multi-service compose topology (existing compose-based installs are not migrated).
- Automated migration from compose to direct runtime for existing installs.

## Target Architecture

### Shared setup front-end
Both deployment modes share:
1. Hardware detection (GPU, RAM, disk).
2. Deployment choice prompt (native vs. container).
3. Pack selection.
4. Review summary and confirmation.

Container mode uses fixed values for native-only prompts: runner=ollama, port=47950, mode=manual, harness=manual.

### Container execution model
Container mode executes via direct runtime primitives:
1. Detect runtime (`podman` preferred, `docker` fallback) via `shutil.which()`.
2. Locate build context (cwd -> parents[4] editable install -> auto-clone to `~/.cache/agentalloy/repo`).
3. Run container preflight checks (runtime available, build assets present, port free).
4. Build image: `{runtime} build -t agentalloy:local -f Containerfile <context>`.
5. Ensure named data volume exists: `{runtime} volume create agentalloy-data`.
6. Create `~/.ollama` directory if it doesn't exist (for models bind mount).
7. Generate entrypoint wrapper script to a temp file (bootstrap logic).
8. Replace existing container if present: `{runtime} run --replace -d --name agentalloy --volume agentalloy-data:/app/data --volume ~/.ollama:/root/.ollama --env AGENTALLOY_PACKS={packs} -p 47950:47950 -e ENTRYPOINT=/tmp/entrypoint.sh agentalloy:local`.
9. Wait for health endpoint: poll `http://localhost:{port}/health` (300s timeout, exponential backoff).
10. Record state to install_state.
11. Run verify step.

No compose orchestration, no sidecar services, no project name / network alias management.

### In-container bootstrap
The Containerfile CMD starts an entrypoint wrapper script generated by the setup script. The bootstrap logic runs inside the container:
1. Check if Ollama is installed. If not, download and install it.
2. Start `ollama serve` in the background on `127.0.0.1:11434`.
3. Poll `http://127.0.0.1:11434` until Ollama is ready (30s timeout).
4. Check if the embedding model (`qwen3-embedding:0.6b`) is already cached. If not, pull it.
5. Check if `/app/.bootstrap-complete` exists. If yes, skip to step 9.
6. Run migrations if needed (`python -m agentalloy.migrate`).
7. If `AGENTALLOY_PACKS` env var is set, run `install-packs` (before uvicorn starts, no lock conflict).
8. Create `/app/.bootstrap-complete` flag file.
9. Start uvicorn on `0.0.0.0:47950`.
10. Trap SIGTERM for graceful shutdown.

### Ollama installation strategy
The setup script orchestrates Ollama installation inside the container. The entrypoint wrapper script is generated dynamically by the setup script (not committed to the repo) and handles the actual installation at first boot.

**Decision**: The entrypoint wrapper is generated by the setup script and written to a temp file before the container is started. It is not committed to the repo. The bootstrap sequence:
1. Check if Ollama is already installed (`which ollama`). If not, download and install it.
2. Start `ollama serve` in the background on `127.0.0.1:11434`.
3. Poll `http://127.0.0.1:11434` until Ollama is ready (30s timeout).
4. Check if the embedding model (`qwen3-embedding:0.6b`) is already cached (`ollama list | grep qwen3-embedding`). If not, pull it.
5. Check if a state flag file exists (`/app/.bootstrap-complete`). If it does, skip directly to step 7.
6. Run migrations if needed (`python -m agentalloy.migrate`).
7. If `AGENTALLOY_PACKS` env var is set, run `install-packs` (before uvicorn starts, no lock conflict).
8. Create the state flag file (`/app/.bootstrap-complete`).
9. Start uvicorn on `0.0.0.0:47950`.
10. Trap SIGTERM for graceful shutdown.

The entrypoint wrapper is idempotent — on subsequent starts, the state flag file causes it to skip directly to step 9.

### Kuzu Lock Resolution
The Kuzu database is single-writer: if the uvicorn service holds the lock, CLI commands via `podman exec` will fail with `RuntimeError: IO exception: Could not set lock on file`. This affects `reembed`, `install-packs`, and `ingest` — all of which open `LadybugStore` with exclusive access.

**Decision**: Use the stop/restart pattern from `container-kuzu-lock-resolution.md`. A shared helper module (`container_service.py`) provides:
- `is_in_container()` — detects container environment (checks `/.dockerenv` or `/app` directory).
- `stop_service_in_container()` — finds the uvicorn PID, sends SIGTERM, waits up to 15s, escalates to SIGKILL if needed.
- `restart_service_in_container()` — restarts the uvicorn process with the same command line, polls `/health` until ready (30s timeout).

This replaces the one-shot container pattern. The single container stays running; the app process is stopped, the CLI command runs, then the app process is restarted. This is the same pattern already used for native installs (systemd/launchd stop/restart in `reembed/cli.py:491-508`), but adapted for the container process tree.

**Implementation notes**:
- The helper must handle the case where the service is already stopped (no-op stop, no-op restart).
- The helper must handle user interrupt (Ctrl+C) — restart the service in a `finally` block.
- The `--no-restart` flag from `reembed` should suppress the restart step for both native and container modes.
- Apply the container-aware stop/restart to `reembed`, `install-packs`, and `ingest` at minimum.

   - Runtime availability (podman or docker on PATH).
   - Build prerequisites (Containerfile, pyproject.toml, uv.lock).

   - Volume readiness.
5. Container mode must produce one endpoint suitable for harness wiring.
6. Embedder (Ollama) and datastore (Kuzu) must both be internal to the containerized deployment.
7. Setup output must clearly state day-2 operations without compose.
8. Ollama must be installed programmatically through the setup script based on detected hardware.
9. Apple Silicon must be handled specially (brew + ollama-app) but still runs CPU-only.
10. The setup script must generate an entrypoint wrapper script dynamically (not committed to the repo).
11. The entrypoint must use a state flag file (`/app/.bootstrap-complete`) for bootstrap completion tracking.

## Volume Layout
The single container uses two host-mounted volumes:

```
/app/
├── data/                          # Named volume: agentalloy-data
│   ├── ladybug                    # Kuzu database (file-level lock)
│   ├── skills.duck                # DuckDB vector store
│   └── skills/                    # Packed skill definitions
└── .ollama/                       # Bind mount: ~/.ollama (host)
    └── models/                    # Ollama model cache
        ├── download/
        └── manifests/
```

The `agentalloy-data` volume persists Kuzu DB, DuckDB vector store, and packed skills. The `agentalloy-models` volume persists Ollama model cache across container rebuilds.

### State Tracking
The entrypoint uses a flag file (`/app/.bootstrap-complete`) to track bootstrap completion. This file is created after the first successful bootstrap (migrations + model pull + install-packs). On subsequent starts, the presence of this file causes the entrypoint to skip directly to starting uvicorn.

If the container crashes mid-bootstrap, the next start will:
- Re-detect Ollama installation status (no-op if already installed).
- Re-check Ollama health (no-op if already running).
- Re-check model cache (no-op if already cached).
- Re-check the state flag file (not present → re-run migrations and install-packs).

Migration idempotency (`python -m agentalloy.migrate`) means re-running it is safe even if the crash occurred mid-migration. The `install-packs` command should also be idempotent (re-running it should not duplicate packs).

### Volume Strategy
- `agentalloy-data`: Named volume (`podman volume create agentalloy-data`). Managed by the container runtime, easy to clean up.
- `agentalloy-models`: Bind mount to `~/.ollama` on the host. More transparent for model management — users can inspect, delete, or reinstall models directly from the host filesystem. The setup script creates the directory if it doesn't exist.
